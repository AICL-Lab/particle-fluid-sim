# 性能指南

本文档聚焦于当前代码库的实用性能工作。

## 最重要的事项

项目目标是在设备能支持的最高粒子数量下实现流畅的交互体验。最大的性能敏感区域是：

1. **计算通道开销** 在 `compute.wgsl` 中
2. **渲染通道开销** 在 `render.wgsl` 中
3. **CPU 端帧编排** 在 `src/core/renderer.ts` 中
4. **自适应质量选择** 在 `src/core/quality.ts` 中

## 运行时设计选择

| 选择               | 为什么重要                           |
| ------------------ | ------------------------------------ |
| 工作组大小为 `64`  | 保持计算着色器调度与 GPU 架构对齐    |
| 共享着色器前导     | 防止 TypeScript 和 WGSL 之间常量漂移 |
| 持久轨迹纹理       | 避免每帧纹理分配                     |
| 自适应粒子数量     | 防止低端设备运行完整 10K 路径        |
| Uniform 缓冲区复用 | 保持 CPU 端上传小且可预测            |

## 如何测量

### 本地开发

```bash
npm run dev
```

使用应用内置 HUD 观察：

- FPS
- 选定的粒子数量
- 质量等级

### 浏览器工具

1. 打开开发者工具
2. 录制几秒钟的交互
3. 查找长帧、GPU 停顿或重复分配

## 高风险更改

### 着色器更改

- 对计算行为的任何更改都可能影响性能和正确性
- 保持 CPU 参考实现与 WGSL 逻辑对齐
- 在着色器相关更改后重新运行物理和渲染器测试

### 渲染器更改

- 避免在 `renderer.ts` 中进行每帧分配
- 修改轨迹纹理生命周期或通道顺序时要小心
- 除非画布大小改变，否则复用资源

### 质量启发式

- 粒子缩放影响稳定性和用户感知
- 更偏好可预测的上限而非难以推理的巧妙启发式

## 调优检查清单

在合并性能敏感更改之前：

1. 确认更改仍遵循相关的 OpenSpec 要求
2. 检查质量等级或粒子数量是否意外改变
3. 检查是否有资源现在每帧重新创建
4. 重新运行 `npm run verify`
